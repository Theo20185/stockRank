## 1. Phase 2B — Combined-screen stacking → REJECT

**Hypothesis:** stacking value-deep + fundamentalsDirection ≠
declining filters out the worst names without hurting the best,
improving forward returns.

**Implementation:**
- IcObservation gained a `fundamentalsDirection` field computed at
  observation-build time from each snapshot's EPS history + forward
  EPS (mirroring the live ranker's `computeFundamentalsDirection`).
- New `PreDecileFilter` type on `CandidateWeights`. The
  `excludeFundamentalsDirections` field drops matching observations
  before the top-decile cut.
- New `value-deep-no-declining-fundamentals` candidate.

**Decision rule (from roadmap):** adopt only if the candidate
beats unfiltered value-deep by ≥ 1%/yr × 3y AND the dropped-names
cohort underperforms the kept-names cohort.

**Results:**

| Regime | candidate 3y excess vs default | Verdict |
|---|---|---|
| PIT 2018-2023 | **+0.20 pp** (within bootstrap noise) | reject |
| PIT 2010-2018 | **−5.36 pp** (substantially worse) | reject |

### Why the filter hurts in pre-COVID

In a recovery regime, "declining fundamentals" includes companies
emerging from a trough — exactly the value-deep targets that mean-
revert UP. Filtering them out removes some of the engine's best
long-term winners.

This is a clean example of why empirical validation matters. The
filter SOUNDS reasonable ("avoid declining fundamentals") but
actively contradicts the value-deep thesis ("buy temporarily
depressed quality"). The Quality category (10% weight) already
captures profitability indirectly via ROIC + accruals; an explicit
fundamentals-direction screen is redundant on the names that should
pass and harmful on the names value-deep wants to catch.

**Engine change:** none. The `PreDecileFilter` machinery stays in
the codebase as a reusable mechanism for future screen experiments,
but no candidate currently uses it.

**fvTrend filter** (the other half of the original Phase 2B
hypothesis) was deferred — requires backtest-side FV-trend
reconstruction which is a separate spec (H10 still deferred).

## 2. Phase 2D — v2 delisted-name handling → INFRASTRUCTURE BUILT, RECOVERY BLOCKED

**Hypothesis:** the v1 PIT pipeline only audits survivors, so H11
(Quality floor) verdicts are structurally biased — the floor's
biggest job is filtering names that subsequently went bankrupt,
which the survivor-only PIT can't see. v2 adds delisted names back
into each historical universe with realized return = -100% (or
actual takeout price).

**Implementation:**
- `packages/data/src/universe/delisted-symbols.ts` — extracts
  delisted tickers from the Wikipedia changes table, classifies
  removal reasons (bankruptcy / acquired / market-cap-change /
  spinoff / other).
- `--include-delisted` CLI flag on `backtest-ic`. When set with
  `--point-in-time`, attempts to fetch Yahoo chart + EDGAR for each
  delisted symbol and adds successful loads to the working
  universe.

**The wall:**

```
Recovery rates: chart 143/345 (41.4%), EDGAR 0/345 (0.0%), both 0/345
```

Of 345 delisted symbols identified from the changes table:
- **41.4% had recoverable Yahoo chart data** (still trades or
  trades on a different exchange post-delisting).
- **0% had recoverable EDGAR fundamentals.**

Why EDGAR recovery is zero: `fetchCompanyFacts` calls `cikFor` which
reads from `packages/data/src/edgar/cik-lookup.json`, a table
generated by `npm run refresh-cik` from the CURRENT S&P 500 list.
Delisted tickers — even still-trading companies removed for cap
change — aren't in the local lookup. SEC's authoritative
`company_tickers.json` has CIKs for all currently-active filers
(would cover most cap-change removals) but bankruptcies and
acquisitions where the original entity is gone wouldn't be there
either.

**No 0 → 0 means no delisted snapshots got built. The H11/H12
verdicts in this run match the previous PIT 2018-2023 run exactly:**

| Verdict | PIT 2018-2023 (original) | PIT 2018-2023 (Phase 2D, no delisted added) |
|---|---|---|
| H11 | pass (+2.95 pp) | pass (+2.95 pp) |
| H12 | pass (+50.84 pp) | pass (+50.84 pp) |

**The §4 Quality floor decision remains on HOLD** — the v2 audit
that would actually answer the question hasn't run, because the
delisted names couldn't be added to the universe.

### What's needed to unblock the §4 floor decision

Two-step v2.1 work:

1. **Extend `cikFor` with a fallback to SEC's broader filer table.**
   SEC's `company_tickers.json` is updated regularly and lists every
   active filer (~10,000 companies). For currently-trading delisted
   symbols (cap-change removals), this is a one-line fix in the
   `cikFor` lookup: try the local table, fall back to the SEC
   bulk table.
2. **Historical filer index for bankruptcies / acquisitions.** SEC
   keeps EDGAR data for delisted/inactive filers under their old
   CIKs but the active `company_tickers.json` doesn't list them.
   Need either a one-time scrape of EDGAR's full company index
   (`https://www.sec.gov/cgi-bin/browse-edgar?action=getcompany`)
   or a hand-curated mapping for the ~150 most impactful historical
   bankruptcies (LEH, ENE, BS, etc.).

Both are doable. Step 1 is small (1-2 hours) and would recover
~70-80% of cap-change removals. Step 2 is bigger (1-2 days) but
covers the cases that matter most for H11.

**Recommended:** ship step 1 as Phase 2D.1 in a follow-up before
declaring Phase 2 done. Step 2 stays as a Phase 3+ ambition.
